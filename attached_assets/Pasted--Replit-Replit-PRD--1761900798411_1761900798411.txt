产品需求文档（PRD）— Social/Web3/Payment MVP

1. 项目概述
	•	目标：交付一个可上线的最小可用产品（MVP），支持：
	1.	社交账号登录与个人展示页（Showcase）；
	2.	Web3 钱包登录；
	3.	Stripe 支付（一次性与/或订阅，含回调处理与账单状态同步）。
	•	受众：创作者与粉丝用户，PC 与移动端浏览器访问。
	•	上线目的：验证账号体系、付费闭环与基础展示页面的转化。

2. 业务范围（In-Scope）
	1.	账号体系
	•	第三方社交账号登录（至少 1–2 个主流平台）。
	•	登录成功后生成平台账户并可登出。
	2.	Web3 登录
	•	通过钱包签名完成登录与账号绑定。
	•	同一用户可绑定/解绑社交账号与钱包地址。
	3.	支付与结算
	•	Stripe 支付流程（一次性支付或订阅二选一或并存，交付时说明开通范围）。
	•	完成支付后，用户在站内看到清晰的“成功/失败/状态更新”反馈。
	•	后台应能正确同步支付或订阅状态（成功、取消、续订、失败等）。
	4.	Showcase 页面
	•	登录用户可维护公开展示资料（头像、简介、外链等）。
	•	公开访问者可查看 Showcase（无需登录）。
	5.	基础管理
	•	账号与支付记录的基础查询/导出（简单列表/明细即可）。
	•	基础封禁/解禁或可见性开关（最简控台级能力）。

说明：不要求复杂的创作者分账、推荐算法、内容审核与大文件转码等扩展能力，后续迭代。

3. 非范围（Out-of-Scope）
	•	原生 App、推送通知、复杂权益发放/分润、内容审核与大规模内容存储、社交图谱/关注关系、AI/视频转码等。

4. 角色与权限
	•	访客：可浏览公开 Showcase，无法进行账号操作与支付。
	•	注册用户：可登录、绑定/解绑登录方式、编辑个人展示信息、发起支付。
	•	运营/管理员：可浏览基础数据（用户、支付记录）、执行基础可见性/封禁操作。

5. 关键用户流程（高层级）
	1.	社交登录：点击第三方登录 → 授权成功返回站内 → 完成账户创建/绑定 → 进入个人主页。
	2.	Web3 登录：连接钱包 → 签名 → 返回站内完成绑定/登录 → 进入个人主页。
	3.	支付：在站内发起购买或订阅 → 跳转 Stripe 完成付款 → 回站内看到成功页 → 账户状态与后台记录同步更新。
	4.	Showcase 编辑与公开：登录后上传头像/填写简介/添加外链 → 保存 → 对外公开访问页面可见。

6. 第三方集成（业务层面）
	•	社交登录：至少接入 1–2 个主流平台（由 Replit选择优先顺序与配置方式）。
	•	Web3 登录：钱包签名登录（确保可识别并绑定链上地址，避免重放/伪造）。
	•	Stripe：支持站内发起支付，且在支付状态变化时平台能正确更新（需要可靠处理支付完成、失败、取消、续订、退款等事件）。

我们不限制技术框架与部署形态；请自行选择最稳妥的集成路径并在交付文档中说明配置项（仅限产品/运营可读的操作指引级别）。

7. 数据与隐私（高层级要求）
	•	仅收集实现上述功能所需的最小化数据。
	•	提供可导出/删除个人数据的路径（基础版即可）。
	•	敏感信息必须被安全保存，不向前端暴露。
	•	明确数据保存位置与保留周期（交付时说明）。

8. 合规与政策（高层级）
	•	遵循常见隐私与支付合规要求（如 GDPR/CCPA 基本原则、Stripe 合规要求）。
	•	展示或链接隐私政策与服务条款（由我们提供文案，Replit 需支持页面或链接位）。

9. 可靠性与性能目标（不指定实现）
	•	可用性目标：≥ 99.9%（月度）。
	•	回调/支付：支付相关回调需高可靠处理，并具备幂等保护（避免重复记账）。
	•	延迟体验：用户首次打开与完成关键操作的整体体验流畅（无需硬性阈值，交付时提供性能实测说明）。
	•	扩展性：支持从 MVP 并发（百级在线、千级日活）平滑扩展 10 倍，不做停机。

是否使用静态托管/自动伸缩/最小实例数等，由 Replit 自行选择；但需保证回调等关键链路在真实流量下稳定。

10. 安全要求（不指定技术方案）
	•	第三方回调与登录票据需进行来源校验与签名验证。
	•	重要业务（登录、支付）具备基础限速与防滥用措施。
	•	管理入口仅限受控访问，并记录关键操作审计。
	•	对常见安全风险（XSS/CSRF/SQL 注入/重放攻击等）进行基本防护。

11. 可观测性与支持
	•	提供错误日志与事件审计的基本查看方式（用于排障与对账）。
	•	关键链路（登录、支付）出现故障时，可快速定位并恢复（交付时提供排障指引）。
	•	提供“运行健康检查”的说明，便于监控接入。

12. 运行环境与发布（不指定技术栈）
	•	环境：开发、预发布、生产 三套。
	•	生产环境需绑定自定义域名并启用 HTTPS。
	•	提供一份面向运营的“配置清单”：社交登录的回调地址/允许域列表、Stripe 回调地址与密钥配置项等（不需要代码层细节）。

13. 文案与品牌
	•	我方提供 Logo、品牌色与隐私/条款文案。
	•	Replit 交付简洁、易用的 UI（无需复杂设计）。

14. 交付物与验收标准

交付物
	•	可访问的线上站点（生产域名）。
	•	基础后台/面板（或等效能力）：查看用户与支付记录。
	•	面向运营的配置与使用手册（无需技术实现细节）。
	•	简要问题排查指南与联络方式。

验收用例（节选）
	1.	使用两类不同第三方社交账号分别完成注册/登录/登出。
	2.	使用 Web3 钱包完成签名登录与解绑，再次绑定仍可用。
	3.	发起一次性支付或订阅支付 → 支付完成后站内状态正确、后台记录准确。
	4.	模拟支付取消/失败/续订 → 平台状态与记录正确同步。
	5.	编辑 Showcase 并在未登录的浏览器访问到正确的公开页面。
	6.	管理端可查到上述操作生成的数据记录与基础审计信息。
	7.	在轻度压力（并发与事件回调）下功能稳定、无明显失败。

15. 时间与里程碑（建议）
	•	T+0～T+3 天：接入登录（社交 + 钱包）与 Showcase 可用、基础后台可见。
	•	T+4～T+7 天：支付流程打通（含回调同步与对账）、体验优化与验收。
	•	T+8～T+10 天：预发布联调、文档交付、生产切换与稳定观察。

如需更紧/更松排期，可在启动会后根据复杂度共同确认。

16. 约束与假设
	•	我方会及时提供：品牌素材、隐私/条款文案、域名解析与第三方平台账号（社交平台、Stripe）。
	•	任何可能影响支付或登录成功率的限制（地区、风控、KYC 等），请在交付时明确告知并提供替代路径建议。

17. 开放问题（由 Replit 决策/澄清）
	•	登录平台优先顺序与数量（至少 1–2 个）。
	•	支付形态优先级（一次性 vs 订阅）与币种/地区限制。
	•	Showcase 字段的初始范围与可扩展字段建议。
	•	运营后台的最小功能清单（列表/筛选/导出维度）。
	•	部署形态（静态/自动伸缩/实例数/容灾/扩展策略）与成本预估。
