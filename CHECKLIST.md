# 每日工作检查清单

> ProofOfInfluence 4方协作工作流 Checklist

**使用说明**: 每次开始工作时，按照此清单逐项检查

---

## 📅 开始工作前 (5分钟)

### 1. 环境检查
- [ ] 打开 Cursor 编辑器
- [ ] 确认在正确的项目目录 (`ProofOfInfluence`)
- [ ] 拉取最新代码: `git pull origin main`
- [ ] 检查本地开发环境: `npm run dev`
- [ ] 确认数据库连接正常

### 2. 查看更新
- [ ] 检查 GitHub 是否有新的 Issue 或 PR
- [ ] 查看 Replit 部署状态
- [ ] 访问生产 URL 确认服务正常
- [ ] 查看 Replit 日志是否有错误

### 3. 规划今日任务
- [ ] 查看 [COLLABORATION.md](COLLABORATION.md) 确认协作流程
- [ ] 列出今天要完成的 1-3 个任务
- [ ] 确定任务优先级（P0/P1/P2）
- [ ] 评估每个任务所需时间

---

## 💬 与 ChatGPT 规划 (5-10分钟)

### 对于新功能开发
- [ ] 向 ChatGPT 清楚描述功能需求
  ```
  我要为 ProofOfInfluence 添加 [功能名称]
  目标用户: [Web2创作者/投资人/其他]
  核心价值: [1-3个要点]
  约束条件: [时间/技术/其他]
  ```
- [ ] 获得 ChatGPT 的设计方案
  - [ ] 页面结构/组件列表
  - [ ] 数据结构设计
  - [ ] API 接口设计
  - [ ] 技术实现建议
- [ ] 与 ChatGPT 讨论潜在问题和风险
- [ ] 确认实施步骤和时间估算

### 对于 Bug 修复
- [ ] 向 ChatGPT 提供错误信息和上下文
- [ ] 获得诊断分析和解决方案
- [ ] 确认修复方案的风险

---

## 💻 使用 Cursor 开发 (主要工作时间)

### 开始开发前
- [ ] 决定是否需要创建新分支
  - [ ] 大功能: `git checkout -b feat/[功能名]`
  - [ ] 小改动: 直接在 `main` 分支
- [ ] 再次查看 `.cursorrules` 确认编码规范

### 开发过程中
- [ ] 使用 Cursor Composer (Ctrl+I) 生成代码
  - [ ] 提供详细的提示词
  - [ ] 引用相关文件 (`@design_guidelines.md`)
  - [ ] 分步骤实现功能
- [ ] 频繁保存代码 (Ctrl+S)
- [ ] 每完成一个小功能就测试
  - [ ] `npm run dev` 本地运行
  - [ ] 在浏览器测试功能
  - [ ] 测试移动端响应式
- [ ] 检查 TypeScript 类型错误
  - [ ] 查看 Cursor 底部的错误提示
  - [ ] 运行 `npm run check`

### 代码质量检查
- [ ] 删除所有 `console.log` 调试代码
- [ ] 确认没有 `any` 类型（除非必要）
- [ ] 添加必要的 `data-testid` 属性
- [ ] 确认组件有 Props 类型定义
- [ ] 检查代码格式是否统一

---

## 🔍 本地测试 (10-15分钟)

### 功能测试
- [ ] 核心功能正常工作
- [ ] 边界情况处理正确（空数据、错误输入等）
- [ ] 用户交互流畅（点击、输入、滚动）
- [ ] 加载状态显示正常
- [ ] 错误提示友好清晰

### 响应式测试
- [ ] 桌面端 (1920x1080)
- [ ] 平板端 (768x1024)
- [ ] 移动端 (375x667)
- [ ] 切换设备时布局正常

### 性能检查
- [ ] 页面加载速度 < 3秒
- [ ] 无明显的卡顿或延迟
- [ ] 图片正常加载
- [ ] 无内存泄漏（打开 DevTools 检查）

---

## 📝 Git 提交 (2-3分钟)

### 提交前最后检查
- [ ] 再次运行 `npm run dev` 确认正常
- [ ] 查看改动文件: `git status`
- [ ] 查看代码差异: `git diff`
- [ ] 确认没有提交不该提交的文件（.env, node_modules等）

### 提交代码
- [ ] 暂存改动: `git add .` 或 `git add [文件名]`
- [ ] 编写清晰的 commit message
  ```
  类型: 简短描述（50字内）
  
  详细说明（可选）
  
  类型：feat | fix | refactor | style | docs | chore
  ```
- [ ] 提交: `git commit -m "feat: 功能描述"`
- [ ] 推送到 GitHub: `git push origin [分支名]`

---

## 🔀 代码审查与合并 (对于 PR)

### 创建 Pull Request
- [ ] 在 GitHub 上创建 PR
- [ ] PR 标题清晰描述改动
- [ ] 填写 PR 模板
  - [ ] 改动内容
  - [ ] 测试步骤
  - [ ] 截图（如果有 UI 改动）
- [ ] 检查 PR 文件列表

### 审查（可选请 ChatGPT 帮助）
- [ ] 自己审查一遍代码
- [ ] 如有疑问，向 ChatGPT 询问
  ```
  "这段代码有什么问题吗？
  [粘贴代码]"
  ```
- [ ] 确认改动符合预期

### 合并
- [ ] 点击 "Merge pull request"
- [ ] 删除远程分支（如果不再需要）
- [ ] 切回本地 main 分支: `git checkout main`
- [ ] 拉取最新代码: `git pull origin main`

---

## 🚀 部署验证 (3-5分钟)

### 等待 Replit 自动部署
- [ ] 访问 Replit 项目页面

- [ ] 等待部署完成（通常 1-2 分钟）
- [ ] 查看部署日志，确认无错误

### 生产环境测试
- [ ] 访问 Replit 提供的生产 URL
- [ ] 测试新功能是否正常工作
- [ ] 测试关键用户流程
  - [ ] 登录
  - [ ] 创建/编辑链接
  - [ ] 查看公开资料
- [ ] 检查是否有 JavaScript 错误（打开 DevTools Console）

### 部署失败处理
- [ ] 如果部署失败，查看 Replit 部署日志
- [ ] 复制错误信息向 ChatGPT 求助
- [ ] 根据建议修复问题
- [ ] 重新提交并推送

## 🆘 紧急情况处理

### 生产环境出现严重Bug
```
1. [ ] 立即在 GitHub 创建 hotfix 分支
2. [ ] 向 ChatGPT 快速诊断问题
3. [ ] 使用 Cursor 快速修复
4. [ ] 本地测试确认修复有效
5. [ ] 立即提交并推送到 main
6. [ ] 监控 Replit 部署和生产环境
7. [ ] 验证问题已解决
8. [ ] 记录事故和解决方案
```

### 部署持续失败
```
1. [ ] 查看 Replit 详细日志
2. [ ] 复制错误信息
3. [ ] 向 ChatGPT 提供完整上下文
4. [ ] 检查 Replit Secrets 配置
5. [ ] 检查 replit.yaml 配置
6. [ ] 本地运行 npm run build 测试
7. [ ] 如需回滚，使用 git revert
```

## 💡 Tips

### 提高效率
1. ⚡ 早上第一件事：拉取最新代码
2. ⚡ 养成频繁提交的习惯（小步快跑）
3. ⚡ 遇到问题先问 ChatGPT（节省时间）
4. ⚡ 使用 Cursor 快捷键（Ctrl+I 是王道）
5. ⚡ 每天固定时间部署（避免晚上部署）

### 避免常见错误
1. ❌ 不要在没有测试的情况下直接推送
2. ❌ 不要在周五晚上部署大功能
3. ❌ 不要忘记拉取最新代码
4. ❌ 不要提交 .env 文件
5. ❌ 不要在生产环境直接调试

### 4方协作最佳实践
1. 🤝 有疑问先问 ChatGPT（规划师）
2. 🤝 写代码交给 Cursor（执行官）
3. 🤝 部署交给 Replit（运维官）
4. 🤝 决策和验证由人工负责（指挥官）
